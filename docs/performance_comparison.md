# Performance Comparison

This document explains how to compare the two solving modes used by the project.

The program supports two solving strategies:

1. **Normal BFS solve**: performs a fresh breadth-first search for the input state.
2. **Lookup-table solve**: uses a precomputed table containing every reachable state and an optimal next move back toward solved.

## Running the comparison

First build the project:

```bash
make
```

Then run:

```bash
make perf
```

This compares the normal BFS solver against the lookup-table solver if the lookup file exists.

To include the lookup-table mode, build the table first:

```bash
make build-lookup
make perf
```

## Example test position

The default comparison uses the known example scramble:

```text
R D L B R' D' L'
```

Face input:

```text
U: W O W Y B
R: R G Y R O
F: G G B G Y
D: Y W B B G
L: O R O Y W
B: W B O R R
```

Command-line form:

```bash
./build/bin/skewb_solver --solve-any WOWYB RGYRO GGBGY YWBBG OROYW WBORR
```

## Interpretation

Both methods returned the same optimal 7-move solution:

`L D R B' L' D' R'`

The lookup-table method performs fewer logical search attempts, but was slower in this test - likely because of lookup-table loading overhead.

The normal BFS solver is useful because it does not require any precomputed files. However, it repeats the search for every new input.

The lookup-table solver has a larger setup cost because the table must be generated and stored. Once generated, it is much better for repeated solving because each step of the solution is found by lookup rather than by another full search.

The state-space count is also useful for the mathematical analysis of the project. It confirms the number of states generated by the move set and gives the maximum depth, which is the God number for this model and move metric.
